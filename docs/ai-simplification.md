# AI 系统精简方案 — 产品经理决策文档

## 一、现有AI系统分析

### 1.1 当前架构（过度设计）

```
用户需求 → QueryRewrite Agent → 混合检索(BM25+语义+关键词)
    → Goal Agent → Planner Agent (稳妥/集中/探索 3个)
    → Schedule Checker → Critic Agent → 返回结果
```

**每次推荐涉及**：
- 6-8 次 LLM 调用
- 复杂的检索管道（BM25 + 向量语义 + 关键词匹配 + 意图识别 + 画像匹配）
- 多个 Agent 协作（Goal, Planner x3, ScheduleChecker, Critic）
- 每次消耗数千 token

### 1.2 成本估算

| 场景 | LLM调用次数 | 预估token | 单次成本(DeepSeek) |
|------|------------|-----------|-------------------|
| 事件推荐 | 3-4次 | 3000-5000 | ¥0.01-0.02 |
| 计划生成 | 6-8次 | 8000-12000 | ¥0.03-0.05 |
| 自我分析 | 1-2次 | 2000-3000 | ¥0.01 |

**如果有100日活用户，每人每天3次推荐**：
- 月成本 ≈ 100 × 3 × 30 × ¥0.02 = **¥180/月**（可接受）
- 但如果保留计划生成：100 × 1 × 30 × ¥0.05 = **¥150/月**（加上面 = ¥330/月）

### 1.3 现有系统的价值点

**值得保留的**：
1. 混合检索的思路（关键词 + 简单语义匹配）
2. 基于用户画像的推荐逻辑
3. 规则评分作为降级方案
4. 推荐理由生成

**MVP可以砍掉的**：
1. 多Agent协作（Goal/Planner/Critic）
2. 向量语义检索（需要额外的Embedding服务）
3. 计划生成功能（复杂且成本高）
4. 自我分析/智能简历（可以后续迭代）

## 二、精简方案

### 2.1 目标架构

```
用户需求 → 简单query改写 → 关键词检索 + 规则评分
    → 单次LLM调用(排序+理由) → 返回结果
```

### 2.2 保留的核心模块

| 模块 | 策略 | 原因 |
|------|------|------|
| 事件搜索 | 保留现有 JPA Specification | 已验证，性能好 |
| 关键词匹配 | 保留规则评分 | 不需要LLM，零成本 |
| 画像匹配 | 简化为类别+地点偏好 | 核心价值 |
| LLM推荐 | 单次调用，只做排序+理由 | 成本可控 |
| 规则降级 | 保留 | LLM不可用时的兜底 |

### 2.3 砍掉的模块

| 模块 | 原因 | 影响 |
|------|------|------|
| QueryRewrite Agent | 过度工程 | 用简单关键词提取替代 |
| Goal Agent | 计划生成功能砍掉 | 无影响 |
| Planner Agent x3 | 成本太高 | 无影响 |
| Schedule Checker | 依赖计划功能 | 无影响 |
| Critic Agent | 增加延迟和成本 | 推荐质量略降 |
| 向量语义检索 | 需要Embedding服务 | 关键词检索够用 |
| 计划生成 | 成本高、使用率低 | 砍掉整个功能 |
| 自我分析 | 非核心 | 后续迭代 |

### 2.4 简化后的推荐流程

```java
// 伪代码
public RecommendationResult recommend(String userId, String need) {
    // 1. 搜索候选事件（复用现有 JPA Specification）
    List<Event> candidates = eventService.search(keyword, category, location);
    
    // 2. 规则评分（复用现有 ruleMatch 逻辑）
    List<ScoredEvent> scored = candidates.stream()
        .map(e -> ruleScore(e, need, userProfile))
        .filter(s -> s.score >= 50)
        .sorted(byScore)
        .limit(10)
        .toList();
    
    // 3. 单次 LLM 调用（排序 + 生成理由）
    if (llmEnabled && scored.size() > 1) {
        return llmRerank(scored, need, userProfile);
    }
    
    // 4. 降级：直接用规则评分结果
    return ruleBasedResult(scored);
}
```

### 2.5 LLM Prompt 设计（精简版）

```
系统提示：
你是学生成长机会推荐助手。根据学生需求和候选事件，返回推荐排序。
只能推荐候选列表中的事件，不能编造。
输出JSON格式。

用户提示：
学生需求：{need}
候选事件：{top10事件的标题、分类、地点、技能}
学生画像：{已完成数量、偏好类别、偏好地点}

请返回：
{"recommendations": [{"eventId": 1, "score": 85, "reason": "推荐理由"}]}
```

**预估token消耗**：~800-1200（比原来减少70%）

| 模型 | 价格(每千token) | 速度 | 中文能力 | 推荐理由 |
|------|-----------------|------|----------|----------|
| DeepSeek-V3 | ¥0.001 | 快 | 优秀 | 性价比最高 |

### 3.2 推荐方案

**首选：DeepSeek-V3**
- 价格最低（¥0.001/千token）
- 中文能力强
- API兼容OpenAI格式（现有代码几乎不用改）
- 速度快

**备选：通义千问-Turbo**
- 如果DeepSeek不稳定
- 阿里云生态，SLA有保障

## 四、MVP功能取舍

### 4.1 保留（P0）

| 功能 | AI参与度 | 说明 |
|------|----------|------|
| 事件搜索 | 无 | 纯数据库查询 |
| 事件推荐 | 低 | 规则评分 + 单次LLM排序 |
| 规则降级 | 无 | LLM不可用时的兜底 |

### 4.2 砍掉（MVP不做）

| 功能 | 原因 |
|------|------|
| 计划生成 | 成本高、使用率低、复杂度高 |
| 自我分析/智能简历 | 非核心，后续迭代 |
| 多Agent协作 | 过度工程 |
| 向量语义检索 | 需要额外基础设施 |

### 4.3 后续迭代

| 功能 | 优先级 | 触发条件 |
|------|--------|----------|
| 智能简历 | P2 | 用户量>500 |
| 计划生成 | P2 | 用户反馈需求 |
| 向量检索 | P3 | 关键词检索不够用 |

## 五、实施计划

### 5.1 后端改造

1. 新建 `SimpleAiService`（替代现有 `AiService`）
2. 复用现有 `EventService.search()` 和 `ruleMatch()` 逻辑
3. 接入 DeepSeek API（兼容 OpenAI SDK）
4. 保留规则降级方案

### 5.2 前端调整

1. 移除"生成计划"入口
2. 简化AI推荐页面（只保留事件推荐）
3. 保留推荐理由展示

### 5.3 成本预估

| 阶段 | 日活 | 月成本 |
|------|------|--------|
| MVP（你自己用） | 1 | < ¥1 |
| 小规模测试 | 50 | ¥30-50 |
| 正式上线 | 500 | ¥300-500 |

| 决策 | 选择 | 原因 |
